## 一、文档说明

### 1.1 目的
本文档定义「借款业务管理系统（Loan Management System）」的产品目标、用户角色、功能模块、业务规则与验收标准，作为设计、开发与测试的统一依据。

### 1.2 范围
- 内部可控、流程完整、留痕清晰的**非银行场景**借款管理。
- 覆盖：客户、资金方、借款产品、申请审批、合同签署、放款、还款计划、还款登记与客户确认、逾期/展期/重组、台账对账、风控预警、审计日志、客户端门户。

## 二、产品目标与原则

### 2.1 核心目标
1. **资金可追溯**：每笔资金可追溯来源（资金方/入金）、去向（放款/还款分配）。
2. **借款可追溯**：每笔借款可追溯申请、风控、审批、合同、签字、放款、客户确认收款。
3. **还款可追溯**：每笔还款可追溯登记、匹配、客户确认、核销、对应借款与期次。
4. **台账完整**：所有资金往来形成台账，可对账、可导出、可审计。
5. **规则可配置**：利率、费用、逾期费、展期费、分期规则等通过规则引擎/参数配置管理，不写死。
6. **合同与签署留痕**：电子合同、电子签字、签署时间/地点/IP/设备/附件留档。
7. **客户确认闭环**：支持客户确认「已收款」「已还款」「确认本次还款金额与用途」。

### 2.2 产品原则
- 不隐藏任何费用；合同与客户确认页清晰展示应还/已还/费用明细。
- 核心业务数据不物理删除，仅作废/冲正/撤销并留痕。
- 金额变更与关键状态变更必须写审计日志。
- 所有关键业务具备明确状态机。

## 三、用户角色与权限（RBAC）

| 角色编码 | 角色名称 | 职责概要 |
|----------|----------|----------|
| super_admin | 超级管理员 | 全部功能、用户与权限、系统配置 |
| biz_staff | 业务员/客户经理 | 客户维护、发起借款申请、查看合同与放款 |
| risk_staff | 风控专员 | 客户与申请风控审核、预警处理 |
| approver | 审批经理 | 借款审批、展期/重组审批 |
| finance | 财务/出纳 | 入金、放款、还款登记、台账与对账 |
| funder_admin | 资金方管理 | 己方资金账户、入金、对账单、收益查看 |
| legal | 法务/合同管理员 | 合同模板、合同列表与签署流程 |
| collection | 催收/跟进专员 | 逾期列表、客户跟进、还款与展期跟进 |
| client | 客户端用户（借款人） | 合同签署、确认收款/还款、查看借款与应还、上传凭证 |
| auditor | 审计/老板 | 只读报表、驾驶舱、审计日志、导出 |

权限颗粒度：资源 + 操作（如 `loan:create`、`repayment:register`、`contract:sign`），数据范围可通过角色 scope（如资金方 ID）限制。

## 四、功能模块需求概要

### A. 客户管理
- 客户基本信息、身份/KYC、联系方式、紧急联系人、地址/工作/收入。
- 历史借款记录、风险标签、黑名单/观察名单。
- 附件上传（证件、工资单、担保材料等）；列表支持筛选、导出。

### B. 资金来源管理
- 资金方档案、资金账户、入金登记、余额。
- 资金使用去向、每笔放款可关联资金账户/入金。
- 资金方收益统计、对账单、分润结算（逻辑可后续细化）。

### C. 借款产品与规则
- 多产品；期限单位天/周/月、利率/服务费/管理费/逾期费/展期费/违约金可配置。
- 固定/阶梯、按天/周/月计费；提前还款、分期、逾期罚息规则。
- 规则版本化、生效时间；历史合同不受新规则篡改。

### D. 借款申请与审批
- 创建申请（金额、期限、用途、还款方式）；费用试算（规则引擎）。
- 风控审核、多级审批、意见留痕；驳回/补件/重提。
- 审批通过后进入合同阶段。

### E. 合同模板与电子签署
- 模板管理、变量（客户名、金额、期限、利率、费用、签约时间地点等）。
- 自动生成 PDF/HTML；客户在线签字（手写/勾选/验证码）；记录时间、地点、IP、设备、截图。
- 合同版本管理、附件归档。

### F. 放款管理
- 放款单（应放、扣费、实到）；打款账户、收款账户、时间、操作人、凭证。
- 客户确认「已收到款项」；支持分笔放款；放款后自动生成还款计划。

### G. 还款计划
- 自动生成计划表；一次性/分期/提前/部分/展期。
- 每期应还本金/利息/费用/罚息、已还、剩余；支持重算并保留原计划版本。

### H. 还款登记与客户确认
- 财务登记收款（现金/转账/第三方/线下）、凭证上传。
- 系统匹配借款/期次；客户确认「本次还款金额」「对应哪笔借款」；部分还款、争议状态（待确认/已确认/驳回/人工复核）。
- 未经客户确认的还款不自动完结，可走人工复核。

### I. 逾期、展期、分期重组
- 自动识别逾期、规则引擎计算逾期费用；展期/分期重组申请与审批。
- 补充协议签署；新计划生效、原计划版本保留。

### J. 台账与对账
- 借款/放款/还款/资金方/客户往来台账；日资金流水；对账差异预警；导出 Excel/PDF。

### K. 风控与预警
- 逾期/大额/重复借款/身份异常/高风险客户/频繁修改合同预警；人工审核备注。

### L. 审计日志
- 关键操作（创建/审批/签署/放款/登记还款/修改规则）记录；前后值变更；不可物理删除核心数据。

### M. 客户端门户
- 合同查看与签署、借款与应还查询、上传还款凭证、确认收款/还款、消息通知（短信/邮件/站内信/WhatsApp 预留）。

## 六、关键业务规则与约束

### 6.1 借款单反查
任一借款单必须可反查：钱是谁出的、谁批的、谁签的、何时放款、客户何时确认收款。

### 6.2 还款反查
任一还款必须可反查：对应哪笔借款、哪一期、本金/利息/费用/罚息分配、谁登记的、客户是否确认。

### 6.3 状态机
- 借款申请：draft → pending_risk → (risk_rejected | pending_approval) → (rejected | approved) → contracted → disbursed  
- 合同：draft → pending_sign → signed | cancelled  
- 放款：pending → paid → confirmed | cancelled  
- 还款：registered → matched → pending_confirm → (confirmed | rejected | manual_review)  
- 还款计划：active → superseded | completed  

### 6.4 数据一致性
- 放款扣减资金账户余额并写台账；还款核销后更新计划项已还金额并写台账。
- 金额字段统一高精度（Decimal）；关键操作事务内完成。

## 七、非功能需求

### 7.1 性能
- 列表与台账支持分页、筛选；导出异步或限流，避免阻塞。

### 7.2 安全
- 认证：JWT + 刷新机制；客户端与后台可分离或统一。
- 权限：接口与菜单按 RBAC 校验；敏感操作二次确认可选。

### 7.3 兼容性
- 管理后台与客户端门户支持桌面端与移动端响应式或 H5。

### 7.4 部署
- 推荐：Vercel + Supabase 或自建 Node + PostgreSQL；文件存储 Supabase Storage / S3。

## 八、验收标准（概要）

- 流程闭环：申请→审批→合同→放款→客户确认收款→还款计划→还款登记→客户确认→核销，全链路可走通。
- 数据一致：台账与业务单一致；对账无逻辑错误。
- 权限清晰：各角色仅能访问约定菜单与接口。
- 审计完整：关键操作有日志、可导出、可追责。
- 规则可配：上述 12 类规则可通过配置/规则引擎生效，无写死。
