# 借款业务管理系统 Gap Analysis 与 4 期迭代路线

更新时间：2026-03-18

## 1. 评估方法

- 目标基线：PRD（docs/05-PRD-product-requirements.md）与 API 清单（docs/06-API-list.md）
- 现状基线：当前已实现页面与 API 路由
- 结论分级：
  - 已具备：可直接用于业务
  - 部分具备：有入口或核心字段，但流程未闭环
  - 缺失：尚未落地

## 2. 当前能力盘点（基于代码）

### 2.1 已实现（核心骨架）

- 双端登录：管理端 / 客户端
- 门户入口页（角色分流）
- 管理端：工作台、客户登记、费率配置页面
- 客户端：我的借款、合同签署页、还款签署页
- API：
  - 认证：admin login、client login、me
  - 合同：生成、详情、签署
  - 还款：详情、客户确认
  - 仪表盘汇总
  - 费率设置
  - 客户登记

### 2.2 部分具备（需要补全闭环）

- 合同模块：已支持生成/签署，但缺少模板管理、版本管理、作废、PDF 归档闭环
- 还款模块：有确认入口，但缺少还款登记、分配、驳回复核、客户端完整待确认队列
- 仪表盘：已做汇总卡片，缺少趋势图、下钻明细、导出
- 设置中心：目前偏费率，缺少系统级参数中心与通知配置

### 2.3 缺失（完整系统必需）

- 借款申请全流程：创建、提交、风控、多级审批、补件/驳回重提
- 放款全流程：待放款列表、打款、回执、客户确认收款
- 还款计划引擎：计划生成、重算、版本并存
- 逾期/展期/重组：识别、审批、补充协议、新计划替换
- 台账与对账：借款/放款/还款/资金方台账、差异预警
- 风控规则与预警中心
- 审计日志全覆盖
- 用户角色权限管理（角色/权限树/数据 scope）
- 消息通知中心（站内信/短信/邮件）
- 导出中心与任务化导出

## 4. 四期迭代路线（可直接执行）

## Iteration 1（P0，先闭环主流程）

目标：打通“申请 -> 审批 -> 合同 -> 放款 -> 还款计划”主链路。

交付清单：

1. 借款申请 API 与页面
   - /api/loan-applications（增删改查）
   - /api/loan-applications/:id/submit
2. 风控与审批动作
   - /api/loan-applications/:id/risk
   - /api/loan-applications/:id/approve
3. 放款流程
   - /api/disbursements（创建、列表、确认打款）
   - 客户端确认收款
4. 还款计划生成
   - 放款成功后自动生成 repayment plan

验收标准：

- 一笔申请可从创建走到“已放款”
- 放款后自动看到还款计划明细
- 所有关键状态变更写审计日志

## Iteration 2（P0，补还款闭环）

目标：打通“还款登记 -> 分配 -> 客户确认 -> 争议复核 -> 核销”链路。

交付清单：

1. 还款登记与分配
   - /api/repayments
   - /api/repayments/:id/allocate
2. 客户确认与驳回
   - /api/client/repayments/pending-confirm
   - /api/repayments/:id/confirm
   - /api/repayments/:id/reject
3. 人工复核队列
   - /api/repayments/review
   - /api/repayments/:id/review
4. 还款台账
   - 还款流水查询 + 导出

验收标准：

- 还款能精确分配到期次（本金/利息/费用）
- 客户可确认或驳回，驳回进入复核
- 计划项金额与台账一致

## Iteration 3（P1，风控财务能力）

目标：补齐运营管理能力，支持规模化。

交付清单：

1. 逾期识别 + 催收任务
2. 展期/重组申请审批与补充协议
3. 资金方/账户/入金/余额
4. 对账与差异预警
5. Dashboard 趋势图与下钻

验收标准：

- 逾期自动识别并进入任务池
- 展期重组后生成新计划且保留旧版本
- 资金余额、放款、回款在台账可对平

## Iteration 4（P1，治理与企业化）

目标：达到“可审计、可运营、可扩展”的企业级标准。

交付清单：

1. RBAC 完整化（用户/角色/权限树/数据范围）
2. 审计日志中心（查询、实体历史、导出）
3. 消息中心（通知模板、站内信、短信/邮件网关）
4. 导出中心（异步任务、下载记录）
5. 平台可观测性（错误告警、慢查询、关键指标）

验收标准：

- 权限按角色与数据范围生效
- 关键操作可追溯“谁在何时改了什么”
- 高并发导出不阻塞主业务

## 5. MVP 完整系统判定（建议）

满足以下 7 条即可对外宣称“完整系统”：

1. 申请 -> 审批 -> 合同 -> 放款 -> 收款确认 -> 还款 -> 核销全链路可跑通
2. 每笔借款可反查合同、审批、放款、还款与资金来源
3. 任一金额可在台账和业务单据对上
4. 客户争议（还款驳回）有复核闭环
5. 关键状态变更均有审计记录
6. 角色权限与数据范围隔离生效
7. 异常流程（放款失败、对账差异）有可执行处理机制
